### Authentication & authorization

This is a pretty big topic, and one I'm not planning to delve too deep into, this isn't a tutorial as much as it is a rough process map.

But here are some general tips

1. Probably the most important and accurate wisdom I can impart; we developers are bad at security.

Keeping things as short as possible, think of jwt as a secret code, generated by the API and expected in response whenever you want to do something. Imagine one of those underground bars, that have a password that you need to get in. Well, it's essentially thatâ€”the API provides you a password and expects to hear anytime you wanna do stuff, this password is extra special, because it can carry a payload, think data hidden in the token, accessible upon decoding. Here you can put things like user data, that you'd probably need anyway for accurate operation. This is where authentication stops and authorization begins. Now that the API knows itâ€™s you, it can gauge who you are and what you're allowed to do.

This is the big reason routes are split from controllers, as more routes and methods are added, the routes threaten to become too bloated, making it difficult toâ€”for exampleâ€”use them as a reference when working on the front-end. This way the routes stay clear, this coupled with modern Expressâ€™ router lets you create a very coherent map of routes and their various middlewares.
